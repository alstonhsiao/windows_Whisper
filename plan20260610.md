## 一、背景與目的

### 現狀問題
approach-6 目前使用 **Grok STT** 單層轉錄，存在兩個已確認的品質問題：

| 問題 | 症狀 | 根因 |
|---|---|---|
| 字間空格 | `速 度 超 快` | Grok STT 模型行為，無法從 API 層控制 |
| 簡繁混用 | 有時輸出簡體字 | Grok STT 無 `prompt` 欄位，`language: zh-TW` 約束力不足 |

同時，使用者未來需要加入大量**專業術語、客戶名稱、朋友名稱**，而 Grok STT 的 `keyterm` 上限只有 10 個詞。

### 解決方案
加入**第二層 LLM 後處理**：Grok STT → Cerebras（LLM）

```
語音
 ↓
Grok STT（快速轉錄，~1s）
 ↓ 原始文字（可能有空格、簡體）
Cerebras LLM（修正，~170ms）
 → 繁體、無空格、正確標點、正確人名/術語
 ↓
貼上到游標位置
```

### 選擇 Cerebras 的理由
- **速度最快**：TTFT ~80–120ms，輸出速度 ~2,100 TPS（比 GPT-4o-mini 快 8 倍）
- **免費方案**：每天 1M tokens 免費，個人使用綽綽有餘（每次 ~300 tokens × 3,333 次/天）
- **無審查**：美國矽谷公司，跑 Meta Llama / Alibaba Qwen 開源 weights，無中文政治審查
- **詞彙無上限**：system prompt 可放幾百個術語，不受 STT keyterm 10 個限制

### Phase 2：CerebrasProvider 實作（~30 分鐘）

**2-1. 新增 LLM 修正介面**

在 `main.py` Provider 區塊新增：
```python
class LLMCorrectionProvider:
    """LLM 後處理介面。子類實作 correct(text, mode) -> str"""
    name = "base_llm"

    def correct(self, text: str, mode: "Mode") -> str:
        raise NotImplementedError


class CerebrasProvider(LLMCorrectionProvider):
    """Cerebras 快速 LLM 修正（Llama / Qwen）"""
    name = "cerebras"

    def __init__(self, cfg: dict):
        self.cfg = cfg
        self._client = None

    def _get_client(self):
        if self._client is None:
            from cerebras.cloud.sdk import Cerebras
            self._client = Cerebras(api_key=self.cfg["api_key"])
        return self._client

    def correct(self, text: str, mode: "Mode") -> str:
        if not mode.llm_prompt or not text:
            return text
        try:
            client = self._get_client()
            r = client.chat.completions.create(
                model=self.cfg.get("model", "llama3.3-70b"),
                messages=[
                    {"role": "system", "content": mode.llm_prompt},
                    {"role": "user",   "content": text},
                ],
                max_tokens=self.cfg.get("max_tokens", 512),
                temperature=0.0,
            )
            return r.choices[0].message.content.strip()
        except Exception as e:
            print(f"⚠️  Cerebras 修正失敗（{e}），使用原始文字")
            return text  # fallback：返回未修正的文字
```

**2-2. 更新 `load_config()` 和 `build_provider()`**

在 `config.json` 加入 Cerebras 設定區塊：
```json
"llm_correction": {
  "provider": "cerebras",
  "cerebras": {
    "api_key": "",
    "model": "llama3.3-70b",
    "max_tokens": 512
  }
}
```

新增 `build_llm_correction_provider()`：
```python
def build_llm_correction_provider(cfg: dict):
    llm_cfg = cfg.get("llm_correction", {})
    provider_name = llm_cfg.get("provider", "none")
    if provider_name == "none" or not llm_cfg:
        return None
    sub = dict(llm_cfg.get(provider_name, {}))
    if not sub.get("api_key"):
        raise RuntimeError(f"❌ llm_correction.{provider_name} 缺少 api_key")
    return {"cerebras": CerebrasProvider}[provider_name](sub)
```

## 四、最佳實踐

### Prompt 技巧

**1. system prompt 結尾必加**
```
只輸出修正後的文字，不要解釋、不要加「修正後：」之類的前綴。
```
不加這句，模型容易加說明文字，破壞貼上效果。

**2. 詞彙格式用「名稱（備注）」**
```
- 蕭淳云（不是蕭純云、蕭鈞云）
- 加模（合作廠商名稱，不是「家模」）
```
給模型負例，修正率更高。

**3. 口語/正式要明確區分**
- casual mode：不要加句號、保持口語
- pro mode：要加標點、術語英文保留

**4. temperature 固定 0.0**
修正任務不需要創意，固定 0.0 讓輸出穩定可預期。

**5. max_tokens 設 512 就夠**
語音輸入通常 < 200 字，512 tokens 的輸出空間綽綽有餘。

### 工具使用

**新增詞彙的流程**（不需改 code）：
1. 打開 `approach-6-whisper-macos/config.json`
2. 在對應 mode 的 `grok_keyterms` 加詞（重要的，最多 10 個）
3. 在 `llm_prompt` 的詞彙清單加一行說明
4. 存檔 → 重啟程式（`重啟語音輸入.command`）

**測試單一修正（不錄音）**：
